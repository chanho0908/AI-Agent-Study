# 멀티 에이전트 시스템과 Agent 툴

## 왜 하나의 Agent로 부족한가?

단일 Agent는 한계가 있다.

**문제 1: 컨텍스트 한계**  
LLM은 한 번에 처리할 수 있는 텍스트 길이(컨텍스트 창)가 제한되어 있다.  
큰 코드베이스를 전부 읽으면서 분석하면 금방 한도를 초과한다.

**문제 2: 순차 처리의 비효율**  
단일 Agent는 한 번에 하나의 작업만 할 수 있다.  
5개의 파일을 독립적으로 수정해야 한다면, 하나씩 순서대로 처리해야 한다.

**문제 3: 역할 혼재**  
탐색(어디 있지?), 분석(뭐가 문제지?), 수정(어떻게 고치지?)을 같은 Agent가 하면  
각 역할에 최적화된 행동을 하기 어렵다.

**해결책: 멀티 에이전트**

```
단일 Agent:        A → B → C → D (순서대로)

멀티 Agent:   오케스트레이터
              ↙    ↓    ↘
            A     B     C    (병렬 처리)
              ↘    ↓    ↙
              결과 취합 → D
```

## Agent 툴: 서브 에이전트 생성

Claude Code에서 멀티 에이전트는 `Agent` 툴로 구현된다.

```
역할: 새로운 독립적인 에이전트를 생성하고 작업을 위임한다
```

**핵심 개념:**
- 서브 에이전트는 **새로운 대화 컨텍스트**로 시작한다 (부모의 대화 내용을 모름)
- 서브 에이전트는 자신만의 툴을 사용하며 독립적으로 작동한다
- 결과가 나오면 **부모 에이전트에게 단 하나의 메시지**로 반환된다

---

## 오케스트레이터 vs 서브에이전트

멀티 에이전트에는 두 가지 역할이 있다.

### 오케스트레이터 (Orchestrator)

큰 그림을 보고 작업을 분배하는 지휘자.

- 전체 목표를 파악한다
- 작업을 잘게 나눈다
- 각 서브에이전트에게 명확한 지시를 내린다
- 결과를 취합해서 최종 답변을 만든다

### 서브에이전트 (Subagent)

오케스트레이터로부터 특정 작업을 받아 실행하는 전문가.

- 자신에게 주어진 작업에만 집중한다
- 부모의 전체 맥락을 모르므로, 프롬프트가 자기완결적이어야 한다
- 완료되면 결과만 반환한다

## Claude Code의 특수 에이전트 유형

Claude Code는 미리 정의된 특수 목적 에이전트를 갖고 있다.

### Explore — 코드 탐색 전문

```
목적: 코드베이스에서 특정 파일이나 심볼을 빠르게 찾기
특징: 읽기 전용. 쓰기 권한 없음 → 안전하게 탐색에만 집중
사용 예: "이 함수가 어디서 정의됐지?", "이 패턴을 쓰는 파일 다 찾아줘"
```

### Plan — 구현 계획 수립 전문

```
목적: 어떻게 구현할지 설계하기
특징: 코드를 작성하지 않고 계획만 세움
사용 예: "이 기능을 어떻게 추가할지 단계별로 계획해줘"
```

### general-purpose — 범용 에이전트

```
목적: 위 두 유형에 맞지 않는 모든 작업
특징: 모든 툴을 사용할 수 있음
```

## 병렬 처리: 멀티 에이전트의 핵심 장점

```python
# 개념적 예시: 병렬 vs 순차

# 순차 처리 (단일 Agent)
결과1 = analyze_file("auth.py")       # 3초
결과2 = analyze_file("database.py")   # 3초
결과3 = analyze_file("api.py")        # 3초
# 총 9초

# 병렬 처리 (멀티 Agent)
결과1, 결과2, 결과3 = parallel([
    Agent("auth.py 분석해줘"),
    Agent("database.py 분석해줘"),
    Agent("api.py 분석해줘"),
])
# 총 ~3초
```

독립적인 작업들은 동시에 실행하면 시간이 크게 줄어든다.

## 컨텍스트 보호: 또 다른 중요한 장점

서브에이전트는 **새로운 컨텍스트**로 시작하므로, 부모의 컨텍스트를 소비하지 않는다.

```
부모 Agent 컨텍스트 창
┌─────────────────────────────────┐
│ 사용자 대화 (500 토큰)          │
│ 지시 전달 (100 토큰)            │
│ 서브에이전트 결과 요약 (200 토큰)│  ← 결과만 들어옴
└─────────────────────────────────┘

서브에이전트 컨텍스트 창 (별도)
┌─────────────────────────────────┐
│ 지시 (100 토큰)                 │
│ 파일 내용 (2000 토큰)           │  ← 여기서 실컷 씀
│ 분석 중간 과정 (1000 토큰)      │
│ 결과 정리 (200 토큰)            │
└─────────────────────────────────┘
```

큰 작업을 서브에이전트에게 넘기면 부모의 컨텍스트가 깨끗하게 유지된다.

## 좋은 서브에이전트 프롬프트 작성법

서브에이전트는 부모의 대화 내용을 전혀 모른다.  
그래서 프롬프트가 **자기완결적**이어야 한다.

**나쁜 예:**
```
"위에서 말한 파일의 버그를 찾아줘"
→ 서브에이전트는 "위에서 말한 파일"이 뭔지 모름
```

**좋은 예:**
```
"src/auth/login.py 파일에서 JWT 토큰 만료 처리 관련 버그를 찾아줘.
현재 증상: 토큰이 만료됐는데도 로그인이 유지됨.
의심 구역: validate_token() 함수 (120번째 줄 근처).
결과물: 버그 원인과 수정 방법을 코드와 함께 설명해줘."
```

포함해야 할 것들:
- 정확한 파일 경로
- 작업의 배경 (왜 필요한지)
- 어떤 결과를 원하는지
- 참고할 수 있는 힌트

## MCP (Model Context Protocol) — 외부 툴 연결

에이전트 툴의 확장 개념으로, **MCP**가 있다.

```
MCP = 외부 서비스를 Agent의 툴로 연결하는 표준 프로토콜
```

기본 툴(Read, Bash, WebSearch 등)은 빌트인이지만,  
MCP를 통해 제3자 서비스를 Agent의 툴로 추가할 수 있다.

**예시:**

| MCP 서버 | 추가되는 능력 |
|----------|--------------|
| `filesystem` | 특정 디렉토리 접근 권한 관리 |
| `github` | GitHub 이슈/PR 생성, 코드 검색 |
| `slack` | 슬랙 메시지 전송/읽기 |
| `database` | DB 쿼리 실행 |
| `browser` | 웹 브라우저 제어 |

MCP 덕분에 Agent는 거의 모든 외부 서비스와 연동될 수 있다.

## 멀티 에이전트 패턴 3가지

### 패턴 1: 분할 정복 (Divide & Conquer)

```
큰 작업
   ↓ 나눔
A 작업 | B 작업 | C 작업  ← 병렬 실행
   ↓ 취합
최종 결과
```

예시: 대규모 코드베이스 리팩토링 → 파일별로 나눠서 병렬 수정

---

### 패턴 2: 검증자 패턴 (Checker Pattern)

```
작업자 Agent → 결과물
                  ↓
           검증자 Agent → "이 결과 검토해줘"
                  ↓
           승인 or 피드백
```

예시: 코드 작성 Agent + 코드 리뷰 Agent를 따로 둠  
자기 자신의 결과를 검토하는 것보다 독립적인 Agent가 더 객관적

---

### 패턴 3: 전문화 (Specialization)

```
오케스트레이터
├── 탐색 전문 Agent (Explore)
├── 분석 전문 Agent
└── 수정 전문 Agent
```

각 Agent를 특정 작업에 최적화된 시스템 프롬프트로 설정한다.

---

## 정리

| 구분 | 단일 Agent | 멀티 Agent |
|------|-----------|-----------|
| 병렬 처리 | 불가 | 가능 |
| 컨텍스트 한계 | 하나로 제한 | 분산 가능 |
| 역할 전문화 | 어려움 | 쉬움 |
| 복잡도 | 낮음 | 높음 |
| 적합한 작업 | 단순/소규모 | 복잡/대규모 |

> **모든 작업에 멀티 에이전트가 필요하진 않다.**  
> 간단한 작업은 단일 Agent가 훨씬 빠르고 단순하다.  
> 작업의 규모와 독립성을 보고 판단해야 한다.
