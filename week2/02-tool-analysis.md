# 에이전트 툴 분류 및 분석

## 툴이란?

1주차에서 배웠듯이, 툴(Tool)은 Agent가 외부 세계에 영향을 미칠 수 있는 행동 단위다.  
이번에는 Claude Code가 실제로 어떤 툴을 갖고 있는지, 각 툴이 어떤 역할을 하는지 분석한다.

---

## 툴 분류 체계

Claude Code의 툴은 크게 5가지로 분류할 수 있다.

```
에이전트 툴
├── 1. 파일 시스템 툴    → 파일을 읽고 쓰고 수정
├── 2. 실행 툴           → 명령어나 코드를 직접 실행
├── 3. 웹/검색 툴        → 외부 정보를 가져옴
├── 4. 에이전트 툴       → 다른 Agent를 생성하거나 관리
└── 5. UI 관련 툴        → 사용자와 상호작용
```

---

## 1. 파일 시스템 툴

코드를 다루는 Agent에게 가장 기본적인 툴이다.

### Read — 파일 읽기

```
역할: 파일 내용을 읽어서 LLM에 전달한다
특징: 읽기만 하므로 안전. 권한 승인 없이 자동 허용되는 경우가 많음
```

**내부 동작:**
1. LLM이 `Read(file_path="src/main.py")` 요청
2. 하네스가 해당 파일을 열어 내용 반환
3. 반환값이 LLM의 다음 입력으로 들어감

**언제 쓰이나?**
- 버그를 수정하기 전에 현재 코드 확인
- 프로젝트 구조 파악
- 설정 파일 내용 확인

---

### Edit — 파일 수정

```
역할: 파일의 특정 부분을 정확히 교체한다
특징: old_string → new_string 방식. 파일 전체를 덮어쓰지 않음
```

**왜 전체 덮어쓰기가 아니라 부분 교체인가?**  
전체를 덮어쓰면 LLM이 파일 전체 내용을 응답에 포함해야 한다.  
→ 큰 파일에서 토큰 낭비 + 실수 가능성 증가  
부분 교체는 변경이 필요한 부분만 지정하므로 효율적이다.

**조건:** `old_string`이 파일에 정확히 하나만 존재해야 한다.  
두 곳에 같은 문자열이 있으면 오류 → LLM이 더 많은 컨텍스트를 포함해 다시 시도한다.

---

### Write — 파일 생성/전체 재작성

```
역할: 파일을 새로 만들거나 전체를 덮어쓴다
특징: 새 파일 생성에 주로 사용. 기존 파일 수정은 Edit 권장
```

---

### Glob / Grep — 파일 탐색

```
Glob: 패턴으로 파일 목록 찾기 (예: "src/**/*.tsx")
Grep: 파일 내 특정 문자열/패턴 검색
```

파일을 직접 읽기 전에 "어디에 있는지" 파악하는 탐색 툴이다.

---

## 2. 실행 툴

### Bash — 쉘 명령어 실행

Agent의 툴 중 가장 강력하고, 그래서 가장 위험하기도 하다.

```
역할: 임의의 쉘 명령어를 실행하고 결과를 반환
특징: 거의 모든 것을 할 수 있음 → 권한 관리가 중요
```

**Bash 툴이 할 수 있는 것들:**

| 작업 | 예시 명령어 |
|------|-------------|
| 패키지 설치 | `npm install`, `pip install` |
| 테스트 실행 | `pytest`, `npm test` |
| Git 작업 | `git status`, `git commit` |
| 빌드 | `npm run build`, `make` |
| 파일 탐색 | `find . -name "*.py"` |
| 프로세스 관리 | `ps aux`, `kill` |

**왜 이게 위험한가?**  
`rm -rf /`처럼 복구 불가능한 명령어도 실행할 수 있다.  
그래서 Claude Code는 Bash 툴에 대한 세밀한 권한 설정을 지원한다.

```json
"allow": ["Bash(npm run *)"],  // npm run으로 시작하는 명령만 허용
"deny": ["Bash(rm -rf *)"]     // rm -rf는 차단
```

**타임아웃:** 기본 2분. 10분까지 설정 가능.  
장시간 실행되는 명령은 백그라운드로 실행할 수도 있다.

---

## 3. 웹/검색 툴

### WebSearch — 웹 검색

```
역할: 실시간 웹 검색으로 최신 정보를 가져옴
특징: LLM의 학습 데이터 한계를 극복
```

LLM의 지식은 특정 날짜까지만 학습되어 있다 (knowledge cutoff).  
WebSearch는 이 한계를 넘어 최신 정보를 제공한다.

**언제 쓰이나?**
- 최신 라이브러리 API 확인
- 에러 메시지 검색
- 현재 날짜/시간 관련 정보

---

### WebFetch — 특정 URL 내용 가져오기

```
역할: 특정 웹 페이지의 내용을 가져와서 읽음
특징: 공식 문서나 특정 페이지 내용을 직접 참조할 때 사용
```

WebSearch가 "어디에 있나 찾기"라면, WebFetch는 "거기 가서 읽기"다.

---

## 4. 에이전트 툴

### Agent — 서브 에이전트 생성

```
역할: 독립적인 서브 에이전트를 생성해 작업을 위임
특징: 복잡한 작업을 분할하거나 병렬 처리할 때 사용
```

이 툴이 바로 **멀티 에이전트 시스템**의 핵심이다. (3번 문서에서 자세히 다룸)

---

### TodoWrite — 작업 목록 관리

```
역할: 현재 진행 중인 작업을 목록으로 관리
특징: Agent의 "작업 계획 및 진행 상황" 추적
```

LLM이 복잡한 작업을 할 때 할 일 목록을 만들고 완료 여부를 표시한다.  
사용자가 진행 상황을 볼 수 있고, Agent가 작업을 잊지 않게 해준다.

---

## 5. UI/상호작용 툴

### AskUserQuestion — 사용자에게 질문

```
역할: 작업을 진행하기 전에 불명확한 것을 사용자에게 물어봄
특징: Agent가 스스로 판단하기 어려울 때 사용
```

Agent가 무조건 행동하는 게 아니라 **필요할 때 확인하는 것**도 중요한 능력이다.

---

## 툴의 "비용" 비교

모든 툴이 동등하지 않다. 각 툴은 서로 다른 비용과 위험도를 가진다.

| 툴 | 속도 | 위험도 | 역방향 가능? |
|----|------|--------|-------------|
| Read | 빠름 | 없음 | N/A |
| Glob/Grep | 빠름 | 없음 | N/A |
| Edit | 빠름 | 낮음 | 가능 (git restore) |
| Write | 빠름 | 낮음 | 가능 (git restore) |
| Bash | 가변 | **높음** | 불가능할 수 있음 |
| WebSearch | 느림 | 없음 | N/A |
| Agent | 느림 | 가변 | 가변 |

**실전에서의 의미:**  
Claude Code가 파일을 먼저 Read하고 나서 Edit하는 것은 단순한 습관이 아니다.  
하네스가 Edit 전에 Read를 강제하는 규칙이 있고, 이는 잘못된 수정을 최소화하기 위한 설계다.

---

## 툴 선택의 원칙

Agent가 여러 툴 중 어떤 것을 선택하는가?

1. **전용 툴 우선**: 웹 내용이 필요하면 WebFetch > Bash(curl)
2. **안전한 툴 우선**: 파일 수정은 Edit > Bash(sed)
3. **최소 권한 원칙**: 읽기만 필요하면 Bash 대신 Read
4. **병렬 가능 여부**: 독립적인 작업은 여러 툴을 동시에 호출

```
예시: "이 디렉토리에서 TODO 주석 찾아줘"

나쁜 선택: Bash("grep -r TODO . | cat")  // Bash는 강력하지만 과함
좋은 선택: Grep("TODO", path=".")        // 전용 툴이 더 정확하고 안전
```

## 정리

```
파일 시스템 툴  →  코드를 보고 수정하는 눈과 손
실행 툴         →  실제로 뭔가를 실행하는 발
웹 툴           →  인터넷에서 정보를 가져오는 귀
에이전트 툴     →  다른 Agent에게 일을 맡기는 입
UI 툴           →  사용자와 대화하는 목소리
```

툴의 조합이 곧 Agent의 능력이다.  
어떤 툴을 얼마나 잘 설계하느냐가 Agent의 성능을 결정한다.
