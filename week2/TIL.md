# 2주차 TIL — 하네스/에이전트 툴 분석

## 이번 주 가장 중요한 깨달음

**"Agent = LLM + 하네스"**

1주차에는 Agent를 "생각하고 행동하는 AI"로 이해했다.  
이번 주에는 그 구조를 실제로 해부했다.

LLM은 텍스트를 받아서 텍스트를 내는 모델일 뿐이다.  
Agent를 만드는 건 **하네스**다. 하네스가 시스템 프롬프트를 준비하고, 툴을 정의하고, 루프를 돌리고, 결과를 관찰한다.

Claude Code를 예시로 하네스의 구성 요소를 직접 확인할 수 있었다:
- 시스템 프롬프트: 어떻게 행동할지 정의
- 툴 정의: 할 수 있는 행동 목록
- 권한 설정: 안전하게 허용 범위 통제
- 메모리: 파일로 대화 넘어 정보 유지
- 훅: 이벤트 기반 자동화
- 에이전트 루프: LLM-툴 사이의 반복 실행

## 툴에 대한 새로운 시각

툴을 그냥 "Agent가 쓰는 기능" 정도로 생각했는데, 이번에 더 깊이 이해했다.

**툴은 비용이 다르다.**  
Read는 빠르고 안전하다. Bash는 강력하지만 위험하다.  
좋은 Agent는 목적에 맞는 툴을 골라 쓴다.

**툴 설명(description)이 Agent 성능을 결정한다.**  
LLM은 툴의 이름과 설명을 보고 언제 어떤 툴을 쓸지 판단한다.  
설명이 모호하면 잘못된 툴을 쓴다.

**전용 툴이 범용 툴보다 낫다.**  
파일 검색할 때 Bash(grep)보다 Grep 툴이 낫다.  
더 안전하고, 더 명확하고, 더 통제하기 쉽다.

## 멀티 에이전트의 핵심

두 가지 이유로 멀티 에이전트를 쓴다는 걸 이해했다.

1. **병렬 처리**: 독립적인 작업을 동시에 실행 → 속도
2. **컨텍스트 보호**: 서브에이전트가 자기 컨텍스트를 씀 → 부모 컨텍스트 절약

특히 서브에이전트 프롬프트는 자기완결적이어야 한다는 게 중요하다.  
서브에이전트는 부모 대화를 전혀 모르기 때문에, 모든 필요한 정보를 프롬프트에 담아야 한다.

## 3주차 준비: 내가 만들 Agent의 문제 정의

이번 주 학습을 바탕으로 어떤 문제를 Agent로 풀지 후보를 정리했다.

### 후보 1: PR 리뷰 자동화 Agent
- **문제**: 코드 리뷰가 병목. 리뷰어가 같은 지적을 반복함
- **Agent가 할 일**: PR 코드 읽기 → 코딩 컨벤션 체크 → 리뷰 코멘트 생성
- **필요한 툴**: Read, Bash(git), WebFetch(GitHub API)
- **측정 지표**: 리뷰 시간 단축, 동일 지적 반복 횟수

### 후보 2: 일일 업무 정리 Agent
- **문제**: 하루 끝에 오늘 한 일을 정리하는 게 귀찮고 자꾸 빠뜨림
- **Agent가 할 일**: git log, 슬랙 메시지, 캘린더 읽기 → 일일 요약 생성
- **필요한 툴**: Bash(git), MCP(슬랙, 캘린더)
- **측정 지표**: 정리 시간, 빠뜨린 항목 수

### 후보 3: 레거시 코드 분석 Agent
- **문제**: 오래된 코드를 파악하는 데 시간이 너무 오래 걸림
- **Agent가 할 일**: 코드 읽기 → 의존성 파악 → 흐름도 + 요약 문서 생성
- **필요한 툴**: Read, Glob, Grep, Write
- **측정 지표**: 파악 시간, 문서 정확도

3주차에 하나를 골라서 본격적으로 정의하고 설계한다.
